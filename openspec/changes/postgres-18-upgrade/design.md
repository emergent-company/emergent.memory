## Context

The platform currently runs PostgreSQL 17 (`pgvector/pgvector:pg17`). The `memory server upgrade` command already handles automatic in-place major-version upgrades via the `pgautoupgrade` Docker image â€” this mechanism was built and proven for the pg16â†’pg17 migration.

Three constants in `tools/cli/internal/installer/` act as the single source of truth for target version decisions:

| Constant | File | Current value |
|---|---|---|
| `PostgresImage` | `templates.go` | `pgvector/pgvector:pg17` |
| `PostgresMajorVersion` | `templates.go` | `17` |
| `pgUpgradeImage` | `pg_upgrade.go` | `pgautoupgrade/pgautoupgrade:17-bookworm` |

Changing these three constants cascades correctly into all upgrade logic. The Compose templates generated by the installer are derived from `PostgresImage`, so new installs pick up pg18 automatically.

**Alpine client package constraint:** The server runtime image (`alpine:3.21`) only ships `postgresql17-client`. `postgresql18-client` is available from Alpine 3.23 onward. The Dockerfile runtime stage must be bumped to `alpine:3.23` to install the pg18 client tools.

## Goals / Non-Goals

**Goals:**
- New installs use `pgvector/pgvector:pg18`
- `memory server upgrade` on an existing pg17 installation automatically upgrades the data volume to pg18 in-place (same pattern as pg16â†’pg17)
- Server container has pg18-compatible client tools (`pg_isready`, `pg_dump`) via Alpine 3.23
- All static Compose files and Dockerfiles in the repo reference pg18
- E2E CI tests run against pg18

**Non-Goals:**
- Zero-downtime upgrade (brief service stop during `pg_upgrade` is acceptable, consistent with existing behavior)
- Support for upgrading from pg16 directly to pg18 in one step (pg16â†’pg17â†’pg18 is handled by sequential `memory server upgrade` calls; the `pgautoupgrade` image handles this internally anyway)
- Changes to pgvector index types or vector dimensions
- Go driver changes (pgx/v5 and bun are protocol-level and version-agnostic)

## Decisions

### Decision 1: Use `pgautoupgrade` (same as pg16â†’pg17)

**Chosen:** `pgautoupgrade/pgautoupgrade:18-bookworm`

The `pgautoupgrade` image detects the on-disk PG version, installs the appropriate old binaries, runs `pg_upgrade --link` (hard-link mode â€” fast, no data copy), and leaves the data directory in pg18 format. The `PGAUTO_ONESHOT=yes` flag makes it run once and exit.

**Alternatives considered:**
- `pg_dumpall` / restore: simpler mental model but requires proportional time to data size and a separate volume for the dump. pgautoupgrade is faster (hard-link) and proven in this codebase.
- Official `postgres:18` + manual `pg_upgrade`: requires managing two sets of binaries manually. pgautoupgrade abstracts this.
- Skip pg17 and go pg16â†’pg18 in one jump: pgautoupgrade supports multi-hop automatically, but since we already shipped pg17 this is a clean one-hop upgrade.

### Decision 2: Bump runtime to `alpine:3.23`

**Chosen:** `FROM alpine:3.23` in the runtime stage of `deploy/self-hosted/Dockerfile.server`

Alpine 3.21 and 3.22 do not package `postgresql18-client`. Alpine 3.23 does. The bump is minor (same Alpine LTS track, no breaking changes relevant to this image).

**Alternatives considered:**
- Stay on alpine:3.21, install pg client via a custom APK repository: brittle, adds maintenance burden.
- Use the Debian-based pgvector image for the runtime: much larger image, incompatible with existing Alpine-specific tooling.
- Use `pg_isready` from the pg17 client against pg18: `pg_isready` is protocol-compatible across versions and would technically work, but it's better practice to match client version to server version.

### Decision 3: No migration file needed

The upgrade is purely infrastructural â€” no schema changes, no new SQL objects. PG18 is backward-compatible with all existing DDL and pgvector usage (ivfflat indexes, `vector(768)` type, cosine ops).

### Decision 4: Update static Compose files even though installer regenerates them

The static files (`deploy/self-hosted/docker-compose.yml`, `docker/docker-compose.e2e.yml`, etc.) are used directly by developers and CI without going through the installer. They must be updated to pg18 independently of the installer constants.

## Risks / Trade-offs

| Risk | Mitigation |
|---|---|
| `pgautoupgrade:18-bookworm` doesn't exist yet | Confirmed: Docker Hub shows `pgautoupgrade/pgautoupgrade:18-bookworm` is available |
| `pgvector/pgvector:pg18` extension compatibility | Confirmed: `0.8.2-pg18` is published; ivfflat and HNSW indexes are supported |
| Alpine 3.23 introduces a regression in the runtime image | Alpine 3.23 is a stable release (not edge); risk is negligible |
| `pg_upgrade --link` leaves old data files in place (minor) | pgautoupgrade cleans up the old cluster directory automatically |
| Users on pg16 who missed the pg17 upgrade | pgautoupgrade handles multi-hop (pg16â†’pg18 in one run); `detectPostgresVersion` checks `PG_VERSION` file so it will trigger upgrade for any version < 18 |
| Upgrade interrupted midway | Existing stale-lock-file removal logic already handles this; original data is preserved until `pg_upgrade` completes |

## Migration Plan

**New installs:** No action needed. Fresh `memory install` gets pg18 automatically.

**Existing pg17 installations (the common case):**
1. User runs `memory server upgrade`
2. `detectPostgresVersion` reads `PG_VERSION` from the data volume â†’ returns `17`
3. `17 < 18` â†’ `RunPostgresUpgrade` is called
4. `db` service is stopped
5. `pgautoupgrade/pgautoupgrade:18-bookworm` is pulled and run with `PGAUTO_ONESHOT=yes`
6. On success: compose file is updated to `pgvector/pgvector:pg18`, services are restarted
7. On failure: compose file backup is restored, user retains working pg17 instance

**Existing pg16 installations (legacy, < 0.30.x):**
Same flow â€” `16 < 18` triggers upgrade; pgautoupgrade handles multi-hop internally.

**Rollback:**
If `pg_upgrade` fails, the installer already restores the backed-up `docker-compose.yml` so the user can restart their old pg17 instance. The data volume is never destroyed during upgrade.

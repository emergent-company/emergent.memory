## Context

The CLI upgrade commands (`memory upgrade` and `memory server upgrade`) currently fetch the latest GitHub release to determine the target version and download assets. After upgrading, they print a success message but provide no information about what changed between the user's previous version and the new one.

GitHub releases already contain structured "What's Changed" sections with categorized commit summaries (Features, Bug Fixes, Other Changes), generated by CI from commit messages between tags. The GitHub Releases API provides a paginated list endpoint that returns release bodies, which we can use to aggregate changelogs across multiple versions.

Current version tracking:
- **CLI**: Build-time `Version` variable (embedded via ldflags), compared against `release.TagName`
- **Server**: `~/.memory/version` file read via `installer.GetInstalledVersion()`

Both paths know the "from" and "to" versions at upgrade time — we just need to fetch the intermediate releases and display their notes.

## Goals / Non-Goals

**Goals:**
- Show an aggregated changelog after successful `memory upgrade` and `memory server upgrade`
- Fetch release notes from the GitHub Releases API for all versions between old and new
- Parse and display categorized changes (features, fixes, other) grouped by version
- Keep output concise and terminal-friendly (truncate if many releases)
- Fail gracefully — changelog fetch errors must never block the upgrade

**Non-Goals:**
- Authenticating with GitHub (keep using unauthenticated API; 60 req/hr is sufficient for upgrade)
- Showing changelog before upgrade confirmation (only after success)
- Caching release data locally
- Supporting non-GitHub release sources
- Showing server-specific vs CLI-specific changes separately (releases are unified)

## Decisions

### 1. Use the GitHub List Releases API

**Decision**: Call `GET /repos/emergent-company/emergent.memory/releases` (paginated) rather than fetching each release individually.

**Rationale**: A single paginated request (default 30 per page) will cover typical upgrade gaps. Individual requests would require N+1 API calls. The list endpoint returns full release bodies including the "What's Changed" markdown.

**Alternative considered**: Fetching `CHANGELOG.md` from the repo — rejected because the main CHANGELOG only has an `[Unreleased]` section and isn't structured per-release. The CI-generated release notes on GitHub are the canonical per-version changelog.

### 2. Parse release body markdown for the "What's Changed" section

**Decision**: Extract the content between `## What's Changed` and the next `---` separator from each release body. This is the section generated by CI in `.github/workflows/cli.yml`.

**Rationale**: The release body follows a consistent template with "What's Changed" at the top, followed by install instructions separated by `---`. Parsing just this section gives us the categorized changelog (Features, Bug Fixes, Other Changes) without install noise.

**Alternative considered**: Re-categorizing raw commit messages — rejected because CI already does this categorization, and we'd be duplicating logic.

### 3. Display inline in terminal after upgrade success

**Decision**: Print the changelog directly to stdout after the upgrade success message, using simple terminal formatting (headers, bullet points). Limit display to the aggregated changes with version sub-headers.

**Rationale**: Users are already looking at the terminal after upgrade. No need for a pager or separate command for this information. If the gap spans many releases, truncate and point to GitHub releases page.

### 4. Version comparison using semver string sorting

**Decision**: Compare version strings by splitting on `.` and comparing each numeric component. Filter releases whose tag (stripped of `v` prefix) falls strictly between `currentVersion` and `targetVersion`.

**Rationale**: All releases follow `vX.Y.Z` tagging. Full semver library is overkill — simple numeric comparison of 3 components is sufficient.

### 5. Best-effort with graceful degradation

**Decision**: Wrap changelog fetching in a function that returns `(string, error)`. If it fails, log a single-line warning and continue. The upgrade is already complete by this point.

**Rationale**: The changelog is informational only. Network issues, rate limits, or API changes must never cause the upgrade to report failure.

## Risks / Trade-offs

- **[GitHub API rate limit]** → Unauthenticated limit is 60 req/hr. A single paginated request per upgrade is well within this. Mitigation: best-effort, skip on 403/429.
- **[Release body format changes]** → If CI template changes, parser may extract wrong section. Mitigation: if "What's Changed" header not found, show the full body truncated, or skip that release.
- **[Large version gaps]** → User upgrading from very old version could span 50+ releases. Mitigation: cap displayed releases (e.g., 10 most recent), show count of skipped older releases, link to GitHub releases page.
- **[Pre-release / draft releases in API response]** → Mitigation: filter to only non-draft, non-prerelease entries.
